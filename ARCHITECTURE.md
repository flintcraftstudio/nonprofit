# Architecture Guide

This document explains how the advanced template is put together and *why* — the decisions behind the structure, so that anyone (human or agent) starting a client project from it can extend it without reverse-engineering intent. For setup and usage, see [README.md](README.md).

## Purpose & Positioning

This template is the starting point for **Advanced tier** Firefly client projects — anything beyond a static brochure site: authenticated admin areas, CMS-style content, invoicing, data-driven features. It is a superset of the standard tier template, adding:

- A persistence layer (SQLite + goose migrations + sqlc)
- Session-backed authentication with a working login flow
- A reusable middleware library (logging, CORS, rate limiting, CSRF, panic recovery)

The guiding philosophy is **boring, inspectable infrastructure**: standard library first, no web framework, no ORM, vendored frontend JS, a single deployable binary. Every dependency earns its place.

## Stack at a Glance

| Layer | Choice | Why |
|---|---|---|
| Router | Go 1.25 stdlib `net/http` | Method-aware patterns (`GET /login`) made third-party routers unnecessary |
| Templates | [templ](https://templ.guide) | Typed Go components, compile-time errors, agent-friendly |
| Styling | Tailwind standalone CLI | No Node toolchain in the build; binary downloaded by Mage |
| Interactivity | htmx + Alpine.js (vendored in `web/static/js/`) | Server-rendered partials over SPA complexity; no CDN dependency |
| Database | SQLite via `modernc.org/sqlite` | Pure Go (no CGo), zero-ops, single file. [PostgreSQL upgrade path](POSTGRES.md) when a project outgrows it |
| Migrations | goose | Plain SQL files, embedded in the binary, applied automatically at startup |
| Queries | sqlc | Type-safe generated code from SQL in `queries/`, wrapped by `internal/store` |
| Auth | bcrypt + DB-backed sessions | Sessions survive restarts; no JWT complexity |
| Email | Postmark (hand-rolled client) | One endpoint needed; a full SDK isn't justified |
| Spam protection | Cloudflare Turnstile | Contact form verification, degrades gracefully when unconfigured |
| Build | Mage | Build logic in Go, cross-platform, no Makefile dialect |
| Deploy | Docker → GHCR → Hetzner VPS behind Caddy | Push to `main` builds and ships via GitHub Actions |

## Directory Layout

```
cmd/
  server/        # entry point: env loading, DB setup, migrations, routes, graceful shutdown
  seed/          # CLI to create admin users (no public registration flow)
internal/
  config/        # env-var config loader with defaults
  handler/       # one file per feature (home, contact, auth); constructor-style handlers
  mail/          # minimal Postmark client
  middleware/    # generic, reusable net/http middleware (each with tests)
  session/       # session lifecycle, middleware, RequireAuth, context helpers
  store/         # business logic over the sqlc query layer (hashing, type mapping)
  view/          # templ components; *_templ.go output is gitignored
  db/            # sqlc output (gitignored; regenerated by mage generate and in Docker)
migrations/      # goose SQL migrations, numbered; embed.go embeds them into the binary
queries/         # sqlc query definitions — the single source of truth for SQL
web/static/      # CSS output, vendored JS, images — served from disk at /static/
tailwind/        # Tailwind config + standalone binary (gitignored)
```

### Layering rules

- **Handlers** depend on `store`, `session`, `mail`, and `view`. They parse input, call the store, render templ components. No SQL in handlers.
- **Store** is the only package that calls the query layer. SQL is written in `queries/*.sql` and compiled by sqlc into `internal/db`; the store wraps those generated methods and adds business logic (e.g. bcrypt hashing, so a plaintext password never travels further than the handler→store call). To add a query: write SQL in `queries/`, run `mage generate`, expose a `Store` method.
- **Session** depends on an interface (`session.Store`), not the concrete store — it's portable and testable in isolation.
- **View** is render-only. The few globals it holds (`GtagID`, `PixelID`, `TurnstileSiteKey`, `SiteName`) are set once at startup; templates never read the environment.
- **Middleware** is deliberately generic (`func(http.Handler) http.Handler`, config structs, no project imports) so the package can be copied between projects unchanged.

## Request Lifecycle

```
Caddy (TLS, on the VPS)
  → middleware.Logging        (outermost: one structured JSON line per request)
  → middleware.Recovery       (panic → logged stack trace + 500)
  → middleware.RateLimit      (global: 30/sec, burst 60, per IP)
  → middleware.CSRF           (signed double-submit cookie; validates all mutations)
  → session.Middleware        (cookie → DB lookup → user attached to context, or not)
  → http.ServeMux             (method-aware routes; /login POST has a strict extra limiter,
  → handler                    /admin is wrapped in session.RequireAuth)
```

CORS exists in `internal/middleware/` but is opt-in — a same-origin server-rendered site doesn't need it. Rate limiting trusts `X-Forwarded-For` one proxy deep in production (Caddy) and ignores it in development, where the header would be spoofable.

### htmx interaction pattern

Forms post via `hx-post` with `hx-swap="outerHTML"`. Handlers re-render just the form component with inline errors on failure (`view.LoginForm(...)`, `view.ContactForm(...)`), and on success either swap in a success state or send `HX-Redirect`. Handlers also call `http.Redirect` alongside `HX-Redirect`, so plain non-JS form posts still work.

## Data Layer Decisions

**SQLite by default.** Client projects at this tier rarely need more than one process writing to one database. WAL mode and foreign keys are enabled via PRAGMA at startup. The DB file lives at `DB_PATH` (default `./data/app.db`).

**Migrations are embedded and run at startup.** `migrations/embed.go` embeds the SQL files into the binary; both the server and the seed CLI run `goose.Up` against the embedded FS before doing anything else. A deploy is self-migrating with no separate migration step, and the Docker image ships no SQL files. Mage targets exist for running/rolling back migrations manually in development.

**sqlc is the query layer.** SQL lives only in `queries/*.sql`; sqlc compiles it into `internal/db/` (gitignored — regenerated by `mage generate`, the devcontainer, and the Docker build). `internal/store` wraps the generated `db.Queries` and is what handlers and the session package consume.

## Auth Design

- **No registration flow.** Admin users are created with `mage seed <email> <password>` (or `go run ./cmd/seed`). This is deliberate: advanced-tier sites have a handful of known admins, and omitting registration removes an entire attack surface (account enumeration, email verification, password reset flows).
- **Sessions are rows, not tokens-with-claims.** A 32-byte crypto/rand token (hex) is the primary key of a `sessions` row. Logout deletes the row — instant server-side revocation, which JWTs can't do. Cookie: `session_token`, HttpOnly, Secure, SameSite=Lax, 7-day expiry. Expired rows are swept at startup and hourly.
- **The session middleware never blocks.** It attaches the user to the context when the cookie is valid and silently passes through otherwise. Enforcement is separate: wrap protected handlers (or a sub-mux) in `session.RequireAuth` — `GET /admin` is the working example. Templ components can call `session.FromContext(ctx)` directly (the nav does this for its Admin/Logout links).
- **Login is hardened by default.** Unknown email and wrong password produce the same error message (no account enumeration); a strict per-IP limiter (burst 5, then 1/sec) wraps the POST handler; the form carries a CSRF token via the shared `view.CSRFField` component.

## Views

`internal/view/` holds templ components: `layout.templ` (page shell, tracking pixels, asset includes), `nav`/`footer` partials, and one file per page. Generated `*_templ.go` files are gitignored and produced by `templ generate` (via `mage generate`/`mage build`). `view.SiteName` is a constant overridden per project.

## Build & Development

Mage drives everything; `mage build` = Tailwind CSS → `templ generate` + `sqlc generate` → `go build`. The Tailwind standalone binary is downloaded once by `mage installtailwind` (platform-detected, pinned to v3.4.17). `mage dev` does a full build and runs the server.

The devcontainer installs the full toolchain (Go, templ, goose, sqlc, Mage) so a new machine is productive immediately.

## Deployment

Push to `main` triggers `.github/workflows/deploy.yml`:

1. Build the Docker image (multi-stage: Tailwind + templ + sqlc generation + static Go binary on Alpine, `ENV=production` baked in).
2. Push to GHCR tagged `latest` + commit SHA.
3. Sync `docker-compose.prod.yml` to `/opt/<repo-name>` on the VPS.
4. SSH in and `docker compose -f docker-compose.prod.yml pull && up -d`, pinned to the SHA tag (restarts never silently float to a newer `latest`).

Caddy runs on the host, terminates TLS, and proxies to `127.0.0.1:8080`. The SQLite file lives on the `app-data` named volume at `/data/app.db`, so it survives container replacement. One-time VPS setup: `/opt/<project>/.env` with production values, plus the `VPS_HOST`/`VPS_USER`/`VPS_SSH_KEY` repo secrets.

Configuration comes entirely from environment variables. Integrations degrade gracefully when unconfigured — missing Postmark/Turnstile/analytics keys log a warning and disable the feature, and in development a missing `SESSION_SECRET` falls back to an ephemeral secret, so a fresh clone runs with zero setup. Production is strict: the server refuses to start without a real (≥32-byte) `SESSION_SECRET`.

## Per-Project Checklist

Things to verify or decide when starting a project from this template:

1. **Generate a real `SESSION_SECRET`** (`openssl rand -hex 32`) for the VPS `.env` — production refuses to start without one.
2. **Session token hashing.** Tokens are stored raw in the DB; a DB-file leak means hijackable sessions. If the project's threat model warrants it, store `sha256(token)` as the row ID instead.
3. **Static assets are served from disk** with no cache headers or content hashing. Fine for small sites; consider `embed.FS` + hashed filenames for anything traffic-sensitive.
4. **CORS is opt-in.** Wire `middleware.CORS` only if the project serves cross-origin clients (separate frontend, public API).
5. **Tailwind version is pinned in two places** — `magefile.go` and the `Dockerfile`. Keep them in sync when upgrading.
